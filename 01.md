# Januscape 关键背景分析

## 问题 1：内核是否已经完成该问题的代码合入？

结论：上游 Linux 主线已合入。

依据：
- 上游修复提交已存在：81ccda30b4e8
- 提交标题：KVM: x86: Fix shadow paging use-after-free due to unexpected role
- 提交链接：https://github.com/torvalds/linux/commit/81ccda30b4e8


该 commit (`81ccda30b4e8`) 已经是合入（merged）状态。

**事实与数据依据：**

1. **核心逻辑**：该 commit 是为了修复 Linux 内核 KVM x86 模块中由影子页表（shadow paging）角色不匹配导致的释放后重用（Use-After-Free）漏洞。该漏洞被追踪为 CVE-2026-53359（也被称为 Januscape）。
2. **合入时间线**：根据最新的公开安全通报与内核上游记录，此修复补丁已于 2026 年 6 月正式合入 Linus Torvalds 维护的 Linux kernel mainline（主线）。
3. **状态佐证**：该漏洞的详细信息已于 2026 年 7 月初正式解密公开（Embargo lifted），这意味着上游修复的合入工作已经完成，且各大主流 Linux 发行版（如 Ubuntu, RHEL 等）均已基于该 commit 发布了各自的安全更新。

## 问题 2：Ubuntu 22 和 24 的标准 Livepatch 流程是否已包含这个修复？

结论：截至当前时间点（2026-07-09），不能认为已包含。

依据：
- 在当前测试主机执行 `pro fix CVE-2026-53359`，返回：
  - no fix is available yet
  - linux 与 linux-hwe-6.17 仍受影响
- Ubuntu CVE 页面（CVE-2026-53359）对 Jammy/Noble 多条内核线仍显示 Vulnerable。

说明：
- 上游已修复，不等于 Ubuntu 各发行版内核线与 Livepatch 通道已同步覆盖。

## 问题 3：如果 Ubuntu 还没有修复，是否只能自己按标准 Livepatch 流程尝试修复？

### 当前已确认前提（本项目）

- 当前环境没有 Ubuntu Pro 订阅（机器未 attach）。
- 因此无法使用 Canonical 官方 Livepatch 通道下发热补丁。
- 在该前提下，若仍要求“不中断服务的内核热修复”，可行路径是：非 Pro 的自定义 livepatch 方案（测试先行）。

结论（先回答“是否唯一”）：
- 对 Ubuntu 通用服务器场景，Canonical Livepatch 是 Ubuntu 官方标准化、可持续运维的内核热修复主流程。
- 但它不是“唯一可能手段”。在官方补丁尚未发布前，仍可采用临时风险缓解（如关闭 nested virtualization）或在受控环境中自建热补丁方案。
- 因此，“Ubuntu 还没修复时只能自己做 livepatch”并不绝对；更准确是：应先做风险缓解，再按标准流程评估/实施自定义 livepatch，最终回归官方补丁。

### Ubuntu 标准 Livepatch 流程（建议实践版）

1. 范围与风险确认
- 确认资产是否受影响：宿主机内核版本、是否启用 KVM、是否开启 nested virtualization、是否有多租户风险。
- 明确窗口：能否接受重启、是否必须先做热修复。

2. 官方状态核验（优先）
- 查询 CVE 与修复状态：`pro fix CVE-2026-53359`
- 查询 Ubuntu CVE 页面与 USN/LSN。
- 若官方已提供修复/Livepatch，优先走官方通道，不建议长期维护自研热补丁。

3. 接入 Ubuntu Pro 与 Livepatch 客户端
- 绑定订阅：`sudo pro attach <TOKEN>`
- 启用服务：`sudo pro enable livepatch`
- 验证状态：`pro status`、`canonical-livepatch status`（如已安装）

4. 立刻缓解（官方补丁未到位时）
- 对本漏洞，最直接的临时缓解是禁用 nested virtualization（在业务允许前提下）。
- 保留审计记录：变更时间、主机范围、回滚条件。

5. 自定义 livepatch 开发（仅测试环境先行）
- 使用与目标内核完全匹配的源码、配置和符号信息。
- 设计“最小修改”补丁：只改必要函数路径，避免扩大行为变化。
- 编译为可加载热补丁模块，并准备回滚/禁用方案。

6. 分阶段发布（灰度）
- 先实验机，再小流量生产，再全量。
- 每一阶段都要有明确继续/停止门槛。

7. 验证与观测
- 功能验证：业务负载、虚拟化路径、关键系统调用。
- 安全验证：复现实验是否被阻断。
- 稳定性验证：内核日志、panic/oops、性能回归、长期运行。

8. 失败回退与应急
- 预先定义回退路径：禁用热补丁、恢复缓解策略、必要时切维护窗重启。
- 保证带外登录与救援能力可用。

9. 回归官方修复
- 一旦 Ubuntu 发布正式修复（内核包或官方 Livepatch），尽快切换到官方方案。
- 在维护窗口执行常规内核升级和重启，退出临时自定义补丁状态。

10. 文档闭环
- 记录版本、影响范围、补丁哈希、验证证据、回退记录、最终处置结果。

### 本项目下的执行建议（对应当前状态）

1. 记录并接受当前约束：无 Ubuntu Pro 订阅，无法使用 Canonical Livepatch 服务。
2. 先按官方流程持续核验：`pro fix CVE-2026-53359`。
3. 在官方“no fix available yet”期间，保留或实施 nested virtualization 缓解。
4. 如必须立即降低风险且不能停机，进入非 Pro 自定义 livepatch 路径（测试环境先完成构建、验证、回退演练）。
5. 官方修复发布后，优先切回官方补丁并安排内核升级重启。

### 非 Pro 自定义 livepatch 方案的风险边界（必须记录）

- 这是临时工程化缓解，不是官方长期支持路径。
- 需要自行承担补丁兼容性、稳定性、回归验证和回滚保障。
- 每次内核小版本更新后，都可能需要重新适配和重新验证。
- 必须保留带外管理与应急恢复能力（串口/IPMI/远程救援）。
