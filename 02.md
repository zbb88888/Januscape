# 非 Pro 自定义 Live Patch 最小交付清单与合入核验指南

## A. 非 Pro 自定义 live patch 最小交付清单（必须产物）

适用前提：
- 未接入 Ubuntu Pro，无法使用 Canonical Livepatch 官方补丁分发通道。
- 需要在不中断或少中断前提下进行临时内核风险缓解。

### 1. 需求与范围文档

必须产物：
- 漏洞说明与影响范围（主机清单、内核版本、是否启用 KVM/nested virtualization）
- 目标函数与修改范围说明（最小修改原则）
- 不修复时的风险等级与业务影响

验收标准：
- 能明确回答“修什么、不修什么、影响哪些机器”。

### 2. 可复现实验环境

必须产物：
- 可重复启动的测试环境脚本
- 基线复现记录（未打补丁时行为）
- 系统信息快照（内核版本、配置、模块符号）

验收标准：
- 任一同事可按文档独立复现实验。

### 3. 补丁源码与构建工件

必须产物：
- live patch 源码目录
- 构建脚本
- 版本化产物（模块文件、哈希、构建日志）
- 源码与产物映射关系（commit id -> artifact）

验收标准：
- 在指定内核版本上稳定构建成功，产物可追溯。

### 4. 加载、卸载与回滚脚本

必须产物：
- 加载脚本
- 卸载或禁用脚本
- 失败自动回退策略说明
- 带外恢复预案（串口/IPMI/救援模式）

验收标准：
- 演练至少一次完整回滚并记录结果。

### 5. 功能、安全、稳定性验证报告

必须产物：
- 功能验证：业务主路径回归结果
- 安全验证：漏洞触发路径被阻断或行为变化符合预期
- 稳定性验证：内核日志、panic/oops、性能回归、长稳测试

验收标准：
- 有明确“通过/不通过”判定门槛，不是口头结论。

### 6. 灰度发布与变更记录

必须产物：
- 分批发布计划（测试 -> 小流量 -> 全量）
- 每批次准入/停止条件
- 变更单与执行记录

验收标准：
- 任意批次失败可立即停止并回退。

### 7. 退出临时方案计划

必须产物：
- 官方修复跟踪机制（CVE、USN/LSN、发行版内核更新）
- 切回官方补丁的执行窗口计划
- 临时补丁退场条件与时间点

验收标准：
- 临时方案不会无限期留在生产。

## B. 如何确认 81ccda30b4e8 已经是“上游合入”状态

目标提交：
- https://github.com/torvalds/linux/commit/81ccda30b4e8
- 标题：KVM: x86: Fix shadow paging use-after-free due to unexpected role

### 方法 1：在上游仓库确认提交存在于主线历史

步骤：
1. 打开提交页面，确认仓库为 torvalds/linux 且提交对象存在。
2. 检查页面中的分支/版本线索（例如 master、rc 标签等）。
3. 记录提交完整哈希和页面时间戳作为审计证据。

结论判定：
- 页面存在且显示在主线仓库上下文中，可判定“已进入上游主线历史”。

### 方法 2：命令行做可审计核验（推荐）

示例流程：
1. 获取上游仓库引用
2. 校验提交对象存在
3. 校验是否为目标分支祖先
4. 查询包含该提交的标签

参考命令：

```bash
git clone --filter=blob:none https://github.com/torvalds/linux.git
cd linux
git rev-parse --verify 81ccda30b4e83d8f5cc4fd50503c44e3a33abfeb

git fetch origin master --tags
git merge-base --is-ancestor 81ccda30b4e83d8f5cc4fd50503c44e3a33abfeb origin/master && echo MERGED_TO_MASTER

git tag --contains 81ccda30b4e83d8f5cc4fd50503c44e3a33abfeb
```

判定规则：
- merge-base 返回成功，表示提交已在目标分支历史中。
- tag --contains 有输出，表示至少某些发布标签已包含该提交。

### 方法 3：确认是否已进入“你的发行版内核”

注意：
- 上游已合入，不代表 Ubuntu 当前内核包已发布修复。

建议同时核验：
1. Ubuntu CVE 页面状态
2. pro fix CVE-2026-53359 输出
3. 本机匹配源码或二进制行为对照

示例命令：

```bash
pro fix CVE-2026-53359
uname -r
```

结论判定：
- 若发行版状态仍显示 no fix available yet 或 Vulnerable，应视为“上游已合入，但当前发行版线未完成修复落地”。

## C. 本项目建议执行顺序

1. 先完成 A 节最小交付清单中的 1-5 项。
2. 仅在测试环境推进灰度与回滚演练。
3. 持续跟踪官方修复，具备条件后退场临时补丁。
