# 非 Pro 自定义 Live Patch 最小交付清单与合入核验指南

## A. 非 Pro 自定义 live patch 最小交付清单（必须产物）

适用前提：
- 未接入 Ubuntu Pro，无法使用 Canonical Livepatch 官方补丁分发通道。
- 需要在不中断或少中断前提下进行临时内核风险缓解。

### 1. 需求与范围文档

必须产物：
- 漏洞说明与影响范围（主机清单、内核版本、是否启用 KVM/nested virtualization）
- 目标函数与修改范围说明（最小修改原则）
- 不修复时的风险等级与业务影响

验收标准：
- 能明确回答“修什么、不修什么、影响哪些机器”。

### 2. 可复现实验环境

必须产物：
- 可重复启动的测试环境脚本
- 基线复现记录（未打补丁时行为）
- 系统信息快照（内核版本、配置、模块符号）

验收标准：
- 任一同事可按文档独立复现实验。

### 3. 补丁源码与构建工件

必须产物：
- live patch 源码目录
- 构建脚本
- 版本化产物（模块文件、哈希、构建日志）
- 源码与产物映射关系（commit id -> artifact）

验收标准：
- 在指定内核版本上稳定构建成功，产物可追溯。

### 4. 加载、卸载与回滚脚本

必须产物：
- 加载脚本
- 卸载或禁用脚本
- 失败自动回退策略说明
- 带外恢复预案（串口/IPMI/救援模式）

验收标准：
- 演练至少一次完整回滚并记录结果。

### 5. 功能、安全、稳定性验证报告

必须产物：
- 功能验证：业务主路径回归结果
- 安全验证：漏洞触发路径被阻断或行为变化符合预期
- 稳定性验证：内核日志、panic/oops、性能回归、长稳测试

验收标准：
- 有明确“通过/不通过”判定门槛，不是口头结论。

### 6. 灰度发布与变更记录

必须产物：
- 分批发布计划（测试 -> 小流量 -> 全量）
- 每批次准入/停止条件
- 变更单与执行记录

验收标准：
- 任意批次失败可立即停止并回退。

### 7. 退出临时方案计划

必须产物：
- 官方修复跟踪机制（CVE、USN/LSN、发行版内核更新）
- 切回官方补丁的执行窗口计划
- 临时补丁退场条件与时间点

验收标准：
- 临时方案不会无限期留在生产。

## B. 如何确认 81ccda30b4e8 已经是“上游合入”状态

目标提交：
- https://github.com/torvalds/linux/commit/81ccda30b4e8
- 标题：KVM: x86: Fix shadow paging use-after-free due to unexpected role

### 方法 1：在上游仓库确认提交存在于主线历史

步骤：
1. 打开提交页面，确认仓库为 torvalds/linux 且提交对象存在。
2. 检查页面中的分支/版本线索（例如 master、rc 标签等）。
3. 记录提交完整哈希和页面时间戳作为审计证据。

结论判定：
- 页面存在且显示在主线仓库上下文中，可判定“已进入上游主线历史”。

### 方法 2：命令行做可审计核验（推荐）

示例流程：
1. 获取上游仓库引用
2. 校验提交对象存在
3. 校验是否为目标分支祖先
4. 查询包含该提交的标签

参考命令：

```bash
git clone --filter=blob:none https://github.com/torvalds/linux.git
cd linux
git rev-parse --verify 81ccda30b4e83d8f5cc4fd50503c44e3a33abfeb

git fetch origin master --tags
git merge-base --is-ancestor 81ccda30b4e83d8f5cc4fd50503c44e3a33abfeb origin/master && echo MERGED_TO_MASTER

git tag --contains 81ccda30b4e83d8f5cc4fd50503c44e3a33abfeb
```

判定规则：
- merge-base 返回成功，表示提交已在目标分支历史中。
- tag --contains 有输出，表示至少某些发布标签已包含该提交。

### 方法 3：确认是否已进入“你的发行版内核”

注意：
- 上游已合入，不代表 Ubuntu 当前内核包已发布修复。

建议同时核验：
1. Ubuntu CVE 页面状态
2. pro fix CVE-2026-53359 输出
3. 本机匹配源码或二进制行为对照

示例命令：

```bash

uname -r
```

结论判定：
- 若发行版状态仍显示 no fix available yet 或 Vulnerable，应视为“上游已合入，但当前发行版线未完成修复落地”。

## D. 81ccda30b4e8 作为 live patch 候选的可行性评估（本轮）

### 结论摘要

结论：从“代码变更形态”看，`81ccda30b4e8` 符合 live patch 的基本工程要求；可以作为 livepatch 语义参考来制作补丁，但不能机械直接套用到当前 Ubuntu `kvm` 二进制。

### 为什么说“符合 live patch 要求”

1. 变更范围小且集中
- 只涉及 `arch/x86/kvm/mmu/mmu.c` 中单个函数 `kvm_mmu_get_child_sp()` 的局部逻辑。
- 没有跨多个子系统的大范围联动改动。

2. 不涉及结构体布局和内核 ABI 变化
- 变更是条件判断与局部变量初始化顺序调整。
- 不新增/删除对外符号，不改变接口契约。

3. 语义是“局部防护收敛”
- 通过补充 `role.word` 比较，避免错误复用 shadow page。
- 属于典型可热替换函数修复类型（逻辑分支修正）。

4. 目标函数具备函数级替换条件
- 在当前主机 `kvm` 模块中可见目标函数符号与 `__pfx_` 前缀符号，满足 ftrace/livepatch 替换前提。

### 为什么说“不能机械直接套用”

1. 当前 Ubuntu 内核实现与上游片段有差异
- 本机 `kvm_mmu_get_child_sp` 的机器码快路径与公开上游片段并不一致。
- 说明同名函数在发行版分支上可能已有本地改动或不同补丁组合。

2. helper 依赖多为内部静态上下文
- `kvm_mmu_child_role`、`__kvm_mmu_get_shadow_page` 等在模块内为 static linkage。
- livepatch 实现必须严格对齐本机二进制和类型信息，不能只抄源码文本。

## E. Live Patch 标准/原则与本次 commit 的逐项确认

本节目标：把“能不能做、为什么能做、哪些条件还需要额外收口”逐条写清楚，避免口头判断。

### E.1 通用 live patch 工程标准（Hard Gate）

判定口径：下列 1-8 项中，1-6 为必须满足；7-8 为发布前必须完成。

1. 变更粒度最小化（必须）
- 标准：优先函数级替换；不引入跨子系统联动。
- 本次判定：通过。
- 依据：`81ccda30b4e8` 仅调整 `kvm_mmu_get_child_sp()` 内部逻辑。

2. 不改数据结构布局/对外 ABI（必须）
- 标准：不得改 struct/union 布局、不得新增破坏性接口。
- 本次判定：通过。
- 依据：补丁只改局部变量初始化与条件判断，未改外部接口。

3. 目标函数可定位且可挂接（必须）
- 标准：运行时可定位目标函数，存在 livepatch/ftrace 可替换入口。
- 本次判定：通过。
- 依据：`kallsyms-targets.txt` 中存在 `kvm_mmu_get_child_sp` 及 `__pfx_kvm_mmu_get_child_sp`。

4. 目标函数原型可精确恢复（必须）
- 标准：能确认替换函数签名与调用约定一致。
- 本次判定：通过。
- 依据：`btf-target-lines.txt` 可见 `kvm_mmu_get_child_sp` 的 BTF 函数条目。

5. 外部依赖可在补丁模块中安全解析（必须）
- 标准：替换函数所需 helper/符号在目标内核可访问，或可通过等价逻辑规避不可访问依赖。
- 本次判定：有条件通过。
- 依据：`kvm_mmu_child_role`、`__kvm_mmu_get_shadow_page` 在 BTF/kallsyms 中可见但为 static；实现需按本机可链接策略收口。

6. 并发与一致性语义不回退（必须）
- 标准：不能放宽原有并发/校验语义；修复应是“约束收紧”而非“行为扩张”。
- 本次判定：通过。
- 依据：本次核心是新增 `role.word` 相等条件，属于复用路径收敛。

7. 可回滚性与失效安全（发布前必须）
- 标准：必须具备卸载/禁用路径和失败回退脚本。
- 本次判定：未完成（待实现并演练）。

8. 可验证性（功能/安全/稳定性）（发布前必须）
- 标准：必须有通过/不通过门槛与证据归档。
- 本次判定：未完成（待完成编译、加载、回归、长稳）。

### E.2 本次 commit 代码语义拆解与“可 live patch”条件判断

目标变更（来自 `tests/livepatch/upstream_fix.diff`）：

1. `role` 的计算提前
- 代码变化：`union kvm_mmu_page_role role = kvm_mmu_child_role(...);`
- 条件判断：可 live patch。
- 原因：仅局部变量生命周期调整，不引入 ABI 变化；是函数体内部重排。

2. 复用旧 shadow page 的判定条件增加 `role.word` 对比
- 代码变化：在早返回 `ERR_PTR(-EEXIST)` 条件中新增 `spte_to_child_sp(*sptep)->role.word == role.word`。
- 条件判断：可 live patch。
- 原因：纯逻辑分支收紧，目标是阻断“gfn 相同但 role 不同”导致的错误复用/UAF 风险。

3. 末尾 `kvm_mmu_get_shadow_page(vcpu, gfn, role)` 保持语义主路径
- 代码变化：保留主路径调用，只是 role 的获得时机前移。
- 条件判断：可 live patch。
- 原因：主路径调用模型未改变，属于在原控制流上增加防护门槛。

4. 是否存在“必须改调用方”的隐含要求
- 条件判断：当前未发现必须联动修改调用方的证据。
- 结论：满足函数级热替换的最小闭包条件。

	- `is_large_pte()`
	- `spte_to_child_sp()`

2. 在补丁模块中本地重建最小 helper 集
- 由于 Ubuntu 已安装 headers 未暴露这些内部 helper，补丁中使用“本地最小重建”方案。

3. 将早返回门槛升级为更接近上游的条件
- 当前替换函数在返回 `ERR_PTR(-EEXIST)` 之前，已检查：
	- SPTE 为 MMU-present
	- SPTE 不是 large page
	- child shadow page 可解析
	- child `gfn == gfn`
	- child `role.word == role.word`

这正是本次 commit 的核心修复意图：
- 阻止“GFN 相同但 role 不同”的错误 child page 复用。

### G.3 为什么说“这次比 interim 更精确”

此前版本：
- 无条件走 `kvm_mmu_child_role()` + `__kvm_mmu_get_shadow_page()` 主路径；

## H. 各 OS Livepatch 工具链与“内核小版本强绑定”关系（先记录共识）

本节目的：在讨论“复用已有开源工具还是自建”之前，先把事实层分清楚，避免把内核框架、补丁生成工具、商业分发服务混为一体。

### H.1 三层模型（必须区分）

1. 内核框架层（Kernel Framework）
- 主体：Linux 内核原生 livepatch 子系统（`CONFIG_LIVEPATCH` + dynamic ftrace）。
- 特点：开源、跨发行版共享、负责运行时重定向与一致性状态机。

2. 补丁生成工具层（Build Tooling）
- 代表：`kpatch-build`（历史上广泛使用）、上游内核脚本 `scripts/livepatch/klp-build`（新方向）。
- 特点：把“源码差异”转成可加载 `.ko` livepatch 模块。

3. 商业服务/分发层（Service & Distribution）
- 代表：Canonical Livepatch 服务、Red Hat 订阅分发等。
- 特点：后端批量构建+QA+灰度分发；机器侧通常只负责拉取和应用。

### H.2 发行版/生态的关系定位（工程视角）

1. Ubuntu / Canonical
- 运行时依赖 Linux 原生 livepatch 框架（不是私有内核机制）。
- 公开给用户的是 `pro` 客户端 + Livepatch 服务 + on-prem 分发能力。
- 机器侧管理客户端开源（`ubuntu-pro-client`）；后端补丁生产流水线并非完整公开为“用户自助构建服务”。

2. Red Hat / kpatch
- `kpatch` 是开源工具链，能基于差异生成 livepatch 模块并管理加载。
- `kpatch` 项目已进入 maintenance 模式；新趋势是向上游 `klp-build` 收敛。

3. SUSE / kGraft
- 与 kpatch 在历史上路径不同，但在内核主线层面收敛到统一 livepatch 框架与一致性模型。

结论：
- 大家共享同一内核运行时框架；差异主要在“补丁怎么生成、如何测试、如何分发”。

### H.3 为什么说“和内核小版本是强绑定（硬约束）”

绑定不是单点，而是多维同时成立：

1. `vermagic` / 模块 ABI 绑定
- `.ko` 与目标运行内核的构建标识不匹配，加载直接失败。

2. 符号与对象绑定
- livepatch 运行时按符号名 + 对象名（`vmlinux`/模块）定位目标函数；存在同名符号时还需 `old_sympos` 消歧。

3. 二进制布局绑定
- 即使“源码看起来差不多”，只要发行版补丁、编译器、配置差异导致布局变化，补丁安全性就会变化。

4. 语义绑定
- 非平凡修复常依赖内部 helper、局部结构布局、调用路径时序；这部分通常最容易受小版本差异影响。

因此：
- livepatch 不是“内核大版本兼容”问题，而是“具体构建版本兼容”问题。

### H.4 开源边界（本项目决策相关）

1. 开源
- 内核 livepatch 框架：开源。
- `kpatch` 工具链：开源。
- Ubuntu `pro` 客户端：开源。

2. 非完整公开
- 各厂商后端补丁生产与发布流水线通常不是“完整可自助复刻”的公开 SaaS 工具。

### H.5 对我们项目的直接含义（决策输入）

1. “完全自己造一个 livepatch 平台”不合理
- 内核框架和现成工具已经存在，重复造轮子风险高、收益低。

2. “直接拿某工具一键跨版本复用”也不现实
- 强绑定决定了每个目标内核构建都要做专门验证。

3. 更可行的工程路线
- 复用现有框架与开源工具能力（构建、对比、加载管理）；
- 但补丁实现和验证仍按“目标内核构建”逐版本收敛。

### H.6 本文档后续讨论的决策门槛（先定标准）

在比较“复用现有开源工具 vs 自建局部流程”时，统一按以下门槛评估：

1. 可追溯性：能否明确 `source diff -> ko -> target kernel` 全链路映射。
2. 绑定正确性：能否防止误加载到非目标构建。
3. 语义精确性：是否支持复杂 internal helper/静态符号场景。
4. 可回滚性：是否能在 transition 场景下稳定禁用并卸载。
5. 自动化验证：是否能稳定产出 build/enable/rollback 证据。
6. 维护成本：新增 CVE 时的人力与失败率。

## I. 工具路线讨论纪要（本轮结论）

### I.1 已确认结论

结论：当前阶段不建议先下载一整套其他 OS patch 工具源码，也不建议自建完整 patch 工具平台。

### I.2 三个问题的归纳

1. 是否先下载其他 OS 工具源码对比
- 结论：先不做全量下载。
- 原因：当前信息已足够做架构决策；只有在决定试某个具体工具链时，再做定向、小范围 PoC。

2. 内核自带能力是否足够
- 结论：内核自带 livepatch 框架可用，但它是运行时框架，不是完整生产工具链。
- 运行时框架解决：函数重定向与一致性切换。
- 不直接解决：
	- 源码差异到补丁模块的自动生成
	- 多内核版本批量构建
	- 签名/分发/灰度/回滚编排
	- 质量门禁与回归矩阵

3. 若自己做，核心难点在哪里
- 核心难点不是“产出一个 ko”，而是长期、可控、低风险地交付。
- 主要难点：
	- 与内核小版本强绑定，目标变化即要重验
	- static helper 与非导出符号使自动化常需人工收口
	- 语义与并发一致性验证复杂度高于编译构建
	- transition 与回滚稳定性要求极高
	- 随 CVE 与内核升级，维护成本快速上升

### I.3 当前建议路线

1. 不自建通用 patch 工具平台。
2. 继续复用内核 livepatch 框架与当前已跑通流程。
3. 若要比较开源工具，采用“单工具 + 小范围 + 可回滚”对照 PoC。
4. 进入复用决策前，统一按 H 节门槛评估。
